# UX Review（Task-first Product UX）

## 1) 核心角色與任務

### 角色
- **物業經理**：每日追進度、控風險、跨部門協調。
- **行政**：建檔、補資料、維護名單與聯絡資訊。
- **櫃台**：快速查詢住戶/單位/維修狀態，避免回覆延遲。
- **主管**：看 KPI、例外、風險，不看技術細節。

### 核心任務鏈
`建樓 -> 樓層/單位 -> 住戶 -> 租約 -> 修繕 -> 查詢`

---

## 2) 舊版痛點盤點

### A. 資訊密度與理解成本
1. 頁面多以資料欄位堆疊，缺少「下一步」指引。
2. 工程欄位（如 sortIndex）出現在第一視角，與營運語境脫節。
3. 重要狀態（到期風險、空置、進行中修繕）不夠醒目。

### B. 操作負擔
1. 建立資料後缺少明確導流，使用者不確定該去哪頁。
2. 指派住戶與建立租約常需跨頁反覆切換。
3. Owners / Repairs 雖有功能，但入口認知不明顯。

### C. 錯誤風險
1. 表單錯誤多為被動顯示，缺少行動建議。
2. 大動作（例如刪除）缺乏清楚語意提示。
3. 空狀態只顯示「沒資料」，沒有提供下一步。

### D. 回饋一致性
1. 成功/失敗訊息分散，不易建立使用者信任。
2. 狀態命名偏技術（DRAFT/ACTIVE），欠缺營運語境標示。

---

## 3) 本次 UX 改版總結

### IA（資訊架構）重整
- **Dashboard**：今日待辦、風險、下一步。
- **空間管理**：大樓/樓層/單位/stacking。
- **客戶與合約**：住戶、租約、到期風險。
- **維運管理**：owners、common areas、repairs。
- **報表/匯入**：先做 placeholder 入口，避免資訊迷路。

### 共通頁面模板落地
1. 頁首：目的 + 主要 CTA。
2. 摘要卡：KPI / 風險先看。
3. 主要工作區：列表/表單/流程。
4. 次要資訊：補充內容或延伸操作。
5. 回饋：一致 success/error/empty state。

### 代表性改善（以 Floors 為例）
- 隱藏 sortIndex（完全不露出工程欄位）。
- 用「樓層 / 區域 / 單位數 / 入住概況 / 快速操作」。
- 加入篩選（全部/地下/地上/僅未配置）與樓層搜尋。
- 上方摘要卡：地下層數、地上層數、總樓層、已配置樓層。
- 精靈化文案：使用者先懂任務，不先看到技術結構。

---

## 4) 尚待持續優化
1. 狀態文案（DRAFT/ACTIVE）可逐步改為更完整中文 + 系統狀態對照。
2. 刪除/重大異動可再加入二次確認視窗。
3. 報表/匯入目前為 placeholder，後續可接 KPI 自訂與批次作業。
